# 开发计划管理器

自动生成结构化的开发计划文档，支持任务分解、进度跟踪和统计分析。

## 使用场景

- 新项目启动时制定开发计划
- 功能开发前的任务分解
- 迭代规划和进度管理
- 任务完成后的进度更新

---

## 核心功能

### 1. 创建开发计划

当用户需要创建开发计划时，按以下流程执行：

#### Step 1: 需求收集

询问用户以下信息：

| 信息项 | 说明 | 示例 |
|--------|------|------|
| 项目/功能名称 | 计划的主体 | SVN代码审查系统 |
| 核心需求 | 主要功能点 | AI审查、多语言支持 |
| 技术栈 | 使用的技术 | Python, OpenAI API |
| 时间预期 | 预计工期 | 2周 |

#### Step 2: 任务分解

将需求分解为可执行的Phase和Task：

```
Phase (阶段)
  └── Task (任务)
        └── 子任务 (可选)
```

**分解原则**：
- 每个Phase聚焦一个模块/功能
- 每个Task不超过4小时
- 任务粒度：可独立验证完成

#### Step 3: 生成计划文档

创建 `{项目名}开发计划.md` 文件，包含以下章节：

## 状态标记规范

### 任务状态符号

| 符号 | 状态 | 使用场景 |
|------|------|----------|
| ⬜ | 未开始 | 任务尚未开始 |
| 🔄 | 进行中 | 任务正在执行 |
| ✅ | 已完成 | 任务已完成并验证 |
| ⏸️ | 暂停 | 任务暂时搁置 |
| ❌ | 取消 | 任务不再需要 |
| ⚠️ | 阻塞 | 任务遇到阻碍 |

### Phase 状态颜色

| 颜色 | 状态 |
|------|------|
| 🟡 | 未开始/进行中 |
| 🟢 | 已完成 |
| 🔴 | 有阻塞 |

## 进度更新流程

### 更新单个任务

当用户报告任务完成时：

1. **更新任务状态**
   ```markdown
   | 1.1 | 创建目录结构 | ✅ 已完成 | 0.5h | |
   ```

2. **更新进度统计表**
   - 已完成 +1
   - 未开始 -1
   - 完成率重新计算

3. **更新总进度**
   ```markdown
   总进度: 25% (10/40 任务完成)
   ```

4. **添加完成记录**
   ```markdown
   | 2026-03-20 | Phase 1 | 1.1 | 创建目录结构 | 目录创建成功 |
   ```

### 批量更新

当完成一个Phase时：

1. 更新该Phase所有任务状态
2. 更新Phase标题状态：`🟢 Phase 1: xxx (已完成)`
3. 勾选完成标准
4. 更新统计表

## 查询进度

当用户询问进度时，提供以下信息：

1. **总体进度**：已完成/总任务数，百分比
2. **当前Phase**：正在进行的阶段
3. **进行中任务**：标记为🔄的任务
4. **阻塞项**：标记为⚠️的任务（如有）
5. **预计剩余时间**：基于未完成任务估算

## 工作流示例

### 创建新计划

```
用户: 帮我规划一个用户登录功能的开发计划

执行:
1. 收集需求 → 登录方式、安全要求、UI需求
2. 分解任务 → Phase: 数据库/后端/前端/测试
3. 生成文档 → 用户登录功能开发计划.md
4. 返回文档路径
```

### 更新进度

```
用户: 任务2.1完成了

执行:
1. 定位任务2.1
2. 更新状态 → ✅ 已完成
3. 更新统计表
4. 更新总进度
5. 添加完成记录
6. 返回最新进度
```

### 查询进度

```
用户: 当前进度怎么样

执行:
1. 读取计划文档
2. 解析统计表
3. 汇报: 总进度60%，Phase 2进行中，当前任务2.3
```

---

## 注意事项

1. **文件命名**：使用 `{项目/功能名}开发计划.md` 格式
2. **实时更新**：每次状态变化都要同步更新统计表
3. **验收标准**：每个Phase必须有明确的完成标准
4. **时间估算**：使用小时为单位，单个任务不超过4h
5. **保持简洁**：任务描述简明扼要，不超过20字
